## Corpus Overview

The file contains **120 JSONL records**, each a three-turn chat example:

1. `system` — fixed Bun backend specialist instruction
2. `user` — task prompt (from `prompt.md` + `task.yaml` metadata)

All examples score **100/100**, were produced by **`claude-code`**, and passed
both public and hidden tests at export time.

## Duplicate Rate

Three duplicate notions apply to this corpus:

| Type | Rate | Detail |
| --- | --- | --- |
| **Exact `run_id` duplicates** | **0%** | All 120 lines reference distinct runs |
| **Exact patch duplicates** | **3.3%** | 4 examples share an identical assistant diff with another example (3 duplicate groups) |
| **Multi-run per task** | **60.0%** | 72 of 120 examples are 2nd/3rd successful runs of a task already represented |

The high multi-run rate is structural: the exporter scans all perfect runs under
`runs/**`, so tasks solved multiple times contribute multiple near-identical
training rows. Exact patch duplicates (3.3%) are the tighter deduplication signal.

Fine-tuning on this file alone would be closer to a smoke test than a meaningful
specialization run.

### Scaling estimates

If every `dev` task were deduplicated to one example and all 50 tasks (including
the 2 `private_eval` tasks, excluded from training) contributed one perfect run:

| Scenario | Examples | Est. tokens |
| --- | --- | --- |
| Current (multi-run, no dedup) | 120 | ~214K |
| One example per exported task | 48 | ~85K |
| One example per repo task (excl. `private_eval`) | 48 | ~85K |
| One example per repo task (incl. `private_eval`, not recommended) | 50 | ~89K |

To reach a practical SFT scale (e.g. 1M+ tokens), expect to need either many
more tasks, longer solutions, or deliberate data multiplication with dedup
controls.

## Recommendations

1. **Deduplicate before training** — collapse to one example per `task_id` (or per
   `leakage_group`) to cut the 60% multi-run overlap.
2. **Grow the `train` split** — only 8 examples (~17K tokens) are train-labeled;
   most exported data is `dev`.
3. **Keep `private_eval` excluded** — the exporter's default hygiene is working.
4. **Re-export after more runs** — 2 tasks have no successful exports yet because
   they are `private_eval`, not because of export failures on solved runs.

## Methodology

Analysis performed by parsing `datasets/sft/bun-bench.jsonl` and joining task
metadata from `tasks/*/task.yaml`. Token estimation uses `ceil(char_length / 4)`
over full message content. Percentiles use the nearest-rank method on the sorted
per-example token list.
